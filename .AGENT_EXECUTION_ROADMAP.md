# OpenPawz — Next-Generation Agent Execution Architecture

**Internal Roadmap · Confidential**
**Last Updated:** March 2026
**Status:** Proposed · Phases 0–4

## Executive Summary

OpenPawz already has something no other desktop AI platform has: the **Conductor Protocol** (AI-compiled flow execution), the **Foreman Protocol** (cost-free tool execution via worker models), **Tool RAG** (semantic tool discovery), and **Project Engram** (biologically-inspired memory). These are real, implemented, tested systems — not whitepapers.

But right now, our agents still communicate through **natural language**. When Agent A delegates to Agent B, it writes English. When an agent calls a tool, it generates JSON text, serializes it, sends it over IPC, the engine parses it, calls the API, serializes the result back to JSON, and the agent reads it. Every step is a translation layer. Every translation is latency and lost information.

This roadmap describes how we eliminate those translation layers — **not by inventing a new physics, but by removing waste that the industry accepts as normal.**

The core insight: the bottleneck in agent execution is not how fast models think. It's how many times we make them think. Every unnecessary inference call costs 2–10 seconds. Every unnecessary JSON serialization wastes CPU. Every time we stuff 400 tool schemas into context, we burn thousands of tokens the model has to read before it can act.

**We already solved parts of this.** The Conductor's Collapse primitive merges 3 LLM calls into 1. Extract bypasses the LLM entirely for deterministic nodes. Tool RAG loads only the tools the agent needs. The Foreman delegates mechanical work to cheap models. This roadmap is the next four steps along the same trajectory.

## Why We're Doing This

### The Problem We're Solving

Every AI agent platform today — including ours — follows the same loop:

```
User speaks → Model reads everything → Model thinks → Model outputs text →
Engine parses text → Engine calls API → API returns data →
Engine serializes result → Model reads result → Model thinks again → ...repeat
```

Each cycle takes 2–10 seconds (LLM inference) plus network latency. A 5-step task takes 20–50 seconds. Users notice. Users leave.

### What This Changes

| Metric | Today | After Phase 4 |
|--------|-------|---------------|
| **Avg tool calls per task** | 3–5 sequential | 1 planning call + parallel execution |
| **Tool selection overhead** | ~5,000 tokens of schemas per request | ~800 tokens (Tool RAG, already done) |
| **JSON parse failures** | ~5% of tool calls need retry | 0% (constrained decoding) |
| **Internal IPC format** | JSON text (human-readable) | Binary (machine-optimized) |
| **Multi-step task latency** | 20–50s | 4–12s |
| **Token cost per task** | 100% baseline | ~40% of baseline |

### Why No One Else Has Done This

1. **Vector incompatibility** — Every model has its own internal representation. There's no universal "thought format." The industry uses English as the lowest common denominator because it works everywhere. We don't need to solve this — we need to optimize around it.

2. **Safety legibility** — Regulators and users want to read what the AI is doing. Binary protocols scare people. Our answer: the Foreman already monitors at the API-call level, not the language level. OAuth scopes physically prevent overreach regardless of what agents "say" internally.

3. **Legacy infrastructure** — The internet runs on REST and JSON. Gmail won't expose a "latent vector interface." Slack won't accept tensor embeddings instead of text. We have to meet external systems where they are. But *internally* — between our own engine components — we control both sides of the wire.

4. **It requires the right foundation.** You need the Conductor (flow compilation), the Foreman (cost-tier separation), and Tool RAG (dynamic tool loading) before any of this makes sense. We have all three. That's why we can do this and others can't — they'd have to build the foundation first.

## The Plan

### Phase 0: Action DAG Planning (Single-Shot Execution Plans)

**Risk:** Low — additive feature, no breaking changes
**Dependencies:** None (builds on existing Conductor)

#### What It Is

Today, when an agent has a multi-step task, it executes tools one at a time in a loop:

```
Round 1: Agent thinks → calls tool A → waits for result
Round 2: Agent reads result → thinks → calls tool B → waits
Round 3: Agent reads result → thinks → calls tool C → waits
Total: 3 inference calls, 3 sequential tool executions
```

Action DAG Planning asks the model to output a **complete execution plan** in a single inference call:

```
Round 1: Agent thinks → outputs plan:
  {
    "nodes": [
      { "id": "a", "tool": "google_calendar_list", "args": {...} },
      { "id": "b", "tool": "gmail_search", "args": {...} },
      { "id": "c", "tool": "gmail_send", "args": {...}, "depends_on": ["a", "b"] }
    ]
  }

Engine: executes A and B in parallel → feeds results to C → executes C
Total: 1 inference call, 2 round-trips (parallel + sequential)
```

#### Why It Matters

This is the **single highest-impact optimization** we can make. The dominant cost in any agent task is not tool execution — it's the 2–10 seconds of LLM inference per round. Reducing 5 rounds to 1–2 rounds cuts task latency by 60–80%.

The Conductor already does this for *flows* (pre-designed workflows). Action DAG extends it to *conversations* — the agent itself becomes a mini-compiler, outputting an execution plan instead of making sequential decisions.

#### How We Build It

1. **New system prompt section** — Instruct the model that for multi-step tasks, it can output a `plan` tool call containing a DAG of actions instead of calling tools individually.

2. **Plan parser in Rust** — New module `engine/plan/` that:
   - Validates the DAG (no cycles, all dependencies exist, all tools are valid)
   - Computes execution phases using the same depth-grouping algorithm as the Conductor
   - Executes phases with `tokio::join!` for parallelism

3. **Fallback** — If the model doesn't output a plan (or outputs a malformed one), the existing sequential tool-calling loop continues unchanged. The plan is always additive.

4. **Result injection** — After parallel execution, all results are injected into context for a single follow-up inference call where the agent synthesizes the answer.

#### Parallel Execution Failover

When executing a plan DAG, individual nodes can fail — an API times out, a token expires, a tool returns an error. The system must not abandon the entire plan because one branch failed. Failover strategy:

1. **Per-node error isolation** — Each node in the DAG executes inside its own `tokio::spawn` with a `catch_unwind` boundary. A panic or error in node B does not crash the parallel execution of node A. The failed node produces a structured error result: `{ "status": "error", "node_id": "b", "error": "401 Unauthorized", "retryable": true }`.

2. **Retry with backoff** — If a node fails with a retryable error (network timeout, 429 rate limit, 5xx server error), the executor retries up to 2 times with exponential backoff (1s, 3s). Non-retryable errors (401, 403, 404, schema validation) fail immediately.

3. **Dependency-aware degradation** — If node B fails and node C depends on B, the executor does not attempt C. Instead, C receives a synthetic result: `{ "status": "skipped", "reason": "dependency 'b' failed" }`. Independent nodes (no dependency on B) continue executing unaffected.

4. **Partial result synthesis** — The final synthesis inference call receives all results — successes, errors, and skips. The model can tell the user: "I created the calendar event and sent the invite, but couldn't search your emails because Gmail returned a token error. Would you like me to retry that part?"

5. **Automatic plan splitting** — If the entire parallel phase fails (e.g., network is down), the executor falls back to sequential mode for the remaining nodes. This reuses the existing agent loop — the plan degrades gracefully rather than failing completely.

6. **Timeout per node** — Each node has a configurable timeout (default: 30s for API calls, 60s for MCP tool calls). On timeout, the node is treated as a retryable failure. The overall plan has a hard timeout of 5 minutes, after which all in-flight nodes are cancelled and partial results are synthesized.

```
Plan: A ‖ B ‖ C → D (depends on A, B, C)

Scenario: B fails with 500
  - A: ✅ success
  - B: ❌ retry 1 (1s) → ❌ retry 2 (3s) → ❌ final failure
  - C: ✅ success
  - D: executes with A + C results + B error context
  - User sees: "Completed 3 of 4 steps. Step B (Gmail search) failed — server error."
```

#### Impact

| Before | After |
|--------|-------|
| 5 inference calls for a 5-step task | 1–2 inference calls |
| 20–50s latency | 4–12s latency |
| Sequential tool execution | Parallel where possible |
| ~15,000 tokens consumed | ~5,000 tokens consumed |

#### Files Affected
- `src-tauri/src/engine/plan/` — New module (parser, validator, executor)
- `src-tauri/src/engine/agent_loop/mod.rs` — Detect plan tool call, hand off to plan executor
- `src-tauri/src/engine/chat.rs` — Add plan tool to system prompt for capable models
- `src-tauri/src/engine/tools/mod.rs` — Register plan pseudo-tool

### Phase 1: Constrained Decoding (Zero Parse Failures)

**Risk:** Low-Medium — requires provider-specific integration
**Dependencies:** None (independent of Phase 0)

#### What It Is

Today, when the model calls a tool, it generates JSON freeform. Sometimes it produces:
- A string where an object is expected (we just fixed this in `google_api`)
- Missing required fields
- Extra fields the tool doesn't understand
- Malformed JSON that fails to parse entirely

Constrained decoding forces the model to only generate tokens that produce valid output. The model literally *cannot* produce malformed tool calls because the token sampler only allows tokens that match the tool's JSON schema at each position.

#### Why It Matters

Parse failures cause retry loops. Each retry is another full inference call (2–10s, full token cost). Our current retry rate is ~5% — not catastrophic, but at scale it means 1 in 20 tool calls wastes a full round-trip. For power users running dozens of tool calls per session, this adds up.

More importantly, constrained decoding is the **pragmatic version of "communicating outside language."** We don't skip language — we *restrict* it to only valid utterances. The model's output space shrinks from "any string" to "valid tool call JSON," which is a massive reduction in entropy and error surface.

#### How We Build It

1. **OpenAI:** Already supports `response_format: { type: "json_schema", json_schema: {...} }` and `strict: true` on function calling. We just need to pass our tool schemas in the correct format. Structured Outputs guarantee 100% schema compliance.

2. **Anthropic:** Supports `tool_choice: { type: "tool", name: "..." }` to force a specific tool call. Doesn't have grammar-level enforcement yet, but forced tool choice eliminates the "model decides not to call anything" failure mode.

3. **Google Gemini:** Supports `response_mime_type: "application/json"` with `response_schema`. Can be combined with function declarations for structured tool calls.

4. **Local models (Ollama):** Ollama supports `format: "json"` for all models, and the `llama.cpp` backend supports GBNF grammars for full schema enforcement. We can pass our tool schemas as grammars.

5. **Fallback:** For providers that don't support constrained decoding, we keep the current parsing with retry logic. This is always additive.

#### Impact

| Before | After |
|--------|-------|
| ~5% parse failure rate | 0% for supporting providers |
| Retry loops cost 2–10s each | No retries needed |
| Defensive parsing code throughout | Clean, guaranteed-valid input |
| LLM can hallucinate field names | Schema-enforced field names only |

#### Files Affected
- `src-tauri/src/engine/providers/openai.rs` — Add `strict: true` to function definitions
- `src-tauri/src/engine/providers/anthropic.rs` — Add `tool_choice` modes
- `src-tauri/src/engine/providers/google.rs` — Add `response_schema` support
- `src-tauri/src/engine/tools/mod.rs` — Schema-to-grammar conversion for Ollama
- `src-tauri/src/engine/agent_loop/mod.rs` — Remove retry-on-parse-failure when constrained decoding is active

### Phase 2: Embedding-Indexed Tool Registry (Scale to 100,000 Tools)

**Risk:** Low — extends existing Tool RAG
**Dependencies:** Builds on existing `tool_index.rs` and Engram embedding pipeline

#### What It Is

Tool RAG already handles the "too many tools" problem by letting the agent call `request_tools("send email")` and getting back only the relevant tools. But the current implementation:

- Embeds tool descriptions on-demand (first call builds the index)
- Stores embeddings in memory (lost on restart)
- Searches linearly across all tools
- Doesn't index MCP tools from n8n dynamically

The Embedding-Indexed Tool Registry makes this persistent, fast, and complete:

1. **Persistent vector store** — Tool embeddings stored in SQLite, surviving restarts
2. **Incremental indexing** — When new MCP tools are discovered or community packages installed, only the new tools are embedded
3. **Hierarchical search** — First match against tool *categories* (17 domains), then search within the matched domain. Two-stage retrieval.
4. **MCP auto-index** — Every time `mcp_refresh` discovers new tools, they're automatically embedded and indexed

#### Why It Matters

Today we have ~75 built-in tools + whatever MCP exposes from n8n. As we grow toward hundreds of native tools and thousands of MCP tools, the current system won't scale. Stuffing 1,000 tool descriptions into context is impossible (~200,000 tokens). Even embedding them all on first call takes significant time.

More critically, this is how we handle the "AI communicating outside language" insight correctly. The agent doesn't read tool descriptions in English and pick one — it queries a vector space where tools are represented as embeddings, and the closest match is returned. The "communication" between the agent's intent and the tool registry happens in embedding space, not in natural language. This is already how Tool RAG works; we're making it faster and persistent.

#### How We Build It

1. **New SQLite table** — `tool_embeddings` with columns: `tool_name TEXT PRIMARY KEY, description TEXT, embedding BLOB, domain TEXT, source TEXT ('builtin'|'mcp'|'community'), updated_at INTEGER`

2. **Build-on-startup** — On first launch after update, embed all tools in background (non-blocking). On subsequent launches, only embed tools that changed.

3. **Hierarchical retrieval:**
   ```
   Query: "book a meeting"
   Step 1: Embed query → cosine sim against 17 domain centroids → top domain = "google" (0.87)
   Step 2: Search within "google" domain → [google_calendar_create (0.92), google_calendar_list (0.78)]
   Step 3: Return top-k tools
   ```

4. **MCP hook** — In `registry.rs`, after `register_tools()`, trigger background embedding of new tools.

5. **Deduplication** — If an MCP tool overlaps with a built-in tool (e.g., both can send Slack messages), prefer the built-in (faster, no n8n overhead).

#### Embedding Model Failover (No Local Model Available)

Tool RAG and this registry depend on embeddings — vector representations of tool descriptions. Today, embeddings are generated by a local Ollama model (`nomic-embed-text`). But not every user will have Ollama installed, and some environments (locked-down corporate machines, low-spec hardware) can't run local models at all. The system must work without a local embedding model.

Failover chain (each level falls back to the next):

1. **Tier 1: Local Ollama** (preferred) — `nomic-embed-text` via Ollama. ~50ms per embedding, zero cost, full privacy. This is the default when Ollama is running.

2. **Tier 2: Cloud embedding API** — If Ollama is not available, use the user's configured cloud provider's embedding endpoint:
   - OpenAI: `text-embedding-3-small` (~$0.02 per 1M tokens — tool descriptions are tiny)
   - Google: `text-embedding-004` (free tier available)
   - Anthropic: No embedding API currently — skip to Tier 3
   - The provider is chosen based on the user's configured AI provider in Settings. If they use OpenAI for chat, we use OpenAI for embeddings. No extra configuration needed.

3. **Tier 3: BM25 keyword search** (no embeddings) — If no embedding model is available at all (no Ollama, no cloud API key, or user is offline), the tool registry falls back to **BM25 full-text search** over tool names and descriptions. This is the same BM25 engine Engram already uses for memory search. It's less semantically aware than vector search ("book a meeting" won't match "google_calendar_create" as well), but it still works for exact and partial keyword matches.

4. **Tier 4: Domain-only matching** (absolute fallback) — If even BM25 produces no results, fall back to the domain taxonomy. The `request_tools` query is classified into one of the 17 skill domains by simple keyword rules (e.g., query contains "email" or "mail" → email domain). All tools in that domain are returned. This always works — no model, no index, just string matching.

**Detection and transition:**
- On startup, the engine probes Ollama at `http://127.0.0.1:11434/api/tags`. If it responds, Tier 1 is active.
- If Ollama is absent, the engine checks for a cloud API key. If present, Tier 2 is active.
- If neither is available, Tier 3 (BM25) activates automatically. A non-blocking warning is shown in the UI: "No embedding model available — tool search using keyword matching. For better results, install Ollama or configure a cloud provider."
- If Ollama becomes available later (user starts it mid-session), the engine detects this on the next `request_tools` call and promotes back to Tier 1, triggering a background re-index.

**Cached embeddings survive model absence:**
- Embeddings computed by Tier 1 or 2 are stored in SQLite. If the user previously had Ollama but now doesn't, the cached embeddings are still used for vector search — only *new* tools (from MCP discovery) would fall back to BM25. This means the registry degrades gracefully: existing tools keep vector search quality, only newly discovered tools use keyword fallback until an embedding model is available again.

#### Impact

| Before | After |
|--------|-------|
| In-memory index, rebuilt on restart | Persistent SQLite, instant on startup |
| Linear search across all tools | Hierarchical: domain → tool |
| MCP tools not indexed | Auto-indexed on discovery |
| ~75 tools searchable | 100,000+ tools searchable |
| First `request_tools` call: ~2s (building index) | First call: <100ms (pre-built) |

#### Files Affected
- `src-tauri/src/engine/tool_index.rs` — Persistence layer, hierarchical search, incremental indexing
- `src-tauri/src/engine/sessions/schema.rs` — New `tool_embeddings` table
- `src-tauri/src/engine/mcp/registry.rs` — Auto-index hook after tool discovery
- `src-tauri/src/engine/tools/request_tools.rs` — Use persistent index instead of building on-demand

### Phase 3: Binary IPC for Internal Communication

**Risk:** Medium — touches core IPC boundaries
**Dependencies:** Phase 0 (plan executor needs efficient serialization)

#### What It Is

Right now, every communication between engine components uses JSON:
- Frontend ↔ Rust engine: JSON over Tauri IPC
- Agent loop ↔ Tool executor: Rust structs (already efficient)
- Engine ↔ MCP servers: JSON-RPC (protocol requirement)
- Engine ↔ AI providers: JSON over HTTP (protocol requirement)

We can't change external protocols (Gmail and OpenAI both expect JSON). But we control:
- **Frontend ↔ Engine IPC** — Currently JSON. Can be binary.
- **Tool call serialization** — Currently `serde_json::Value`. Can be zero-copy binary.
- **Agent-to-agent messages** — Currently text stored in SQLite. Can be structured binary.
- **Plan DAG execution results** — Phase 0's parallel results can be binary.

#### Why It Matters

JSON parsing is not the bottleneck for a single tool call. But for high-frequency operations — streaming tokens, rapid tool sequences in a plan DAG, real-time channel bridge messages — the cumulative cost of serialize → transmit → deserialize is significant. Each JSON parse is an allocation. Each serialization is a UTF-8 encoding pass.

Binary formats like FlatBuffers (zero-copy reads) or MessagePack (compact, fast) eliminate these costs:
- **Zero-copy** — The receiver reads the bytes directly without allocating new objects
- **Compact** — 60–80% smaller than equivalent JSON (no key names, no quotes, no whitespace)
- **Type-safe** — The schema is compiled, not parsed at runtime
- **10–50x faster** — No parsing, no UTF-8 decoding, no escape handling

For the plan DAG executor (Phase 0), this matters a lot: when 5 tool results return in parallel and need to be assembled into a single context injection, binary serialization avoids 5 separate parse-serialize cycles.

#### How We Build It

1. **MessagePack for Tauri IPC** — Tauri v2 supports custom serialization. Replace JSON with MessagePack for high-frequency events (streaming tokens, tool results). Keep JSON for infrequent commands (settings, agent CRUD) where human readability helps debugging.

2. **FlatBuffers for plan results** — The plan executor (Phase 0) assembles tool results into a single buffer using FlatBuffers. Zero-copy read when injecting into the next inference context.

3. **Protobuf for agent-to-agent** — When Agent A delegates to Agent B (inter-agent messaging), the message can be a structured Protobuf instead of free text. Both agents' tool schemas are pre-shared, so the message format is known.

4. **Benchmarks first** — Before committing to any format, we benchmark on real workloads. If JSON is fast enough for a given path (e.g., settings CRUD), we keep JSON. Only swap where measurement shows a >5x improvement.

#### Impact

Internally measurable (not user-facing except as latency improvement):

| Path | Before | After |
|------|--------|-------|
| SSE token streaming (high-frequency) | JSON per token | MessagePack batch |
| Tool result assembly (plan DAG) | 5× JSON parse | 1× FlatBuffer read |
| Agent-to-agent delegation | English text | Protobuf struct |
| Frontend event bus | JSON events | MessagePack events |

**Expected latency reduction on multi-tool tasks: 15–30%** (on top of Phase 0's 60–80% reduction from fewer inference calls).

#### Files Affected
- `src-tauri/src/engine/agent_loop/mod.rs` — Binary result assembly
- `src-tauri/src/engine-bridge.ts` → MessagePack serialization for events
- `src-tauri/src/engine/tools/agent_comms.rs` — Structured inter-agent messages
- `src-tauri/src/engine/plan/` — Binary result accumulation
- `Cargo.toml` — New deps: `flatbuffers`, `rmp-serde` (MessagePack)
- `package.json` — New dep: `@msgpack/msgpack`

### Phase 4: Speculative Tool Execution (Branch Prediction for Agents)

**Risk:** Medium-High — requires careful rollback logic
**Dependencies:** Phases 0 + 1 (needs plan DAGs and constrained decoding)

#### What It Is

When the model begins generating a tool call token-by-token, we can often predict the full call before it finishes. For common patterns:

- `google_calendar_list` → next call is probably `google_calendar_create` or `gmail_send`
- `request_tools("email")` → next call is probably `gmail_send`
- `memory_search` → next call is probably `memory_store` or a tool call using the retrieved info

Speculative execution **pre-warms** or even **pre-fires** the next likely API call while the model is still generating:

```
Model is generating: "I'll search your calendar first..."
  → Engine predicts: next tool = google_calendar_list
  → Engine pre-authenticates with Google Calendar API
  → Engine pre-fetches today's events (speculative)

Model finishes: calls google_calendar_list(date: "2026-03-06")
  → Engine: already have the data! Return immediately.
  Saved: 200–800ms of API latency
```

#### Why It Matters

After Phase 0 (plan DAGs), many tasks will still need 2–3 sequential inference calls — the model needs to see results before deciding the next step. Speculative execution hides the API latency of the *next* step behind the inference latency of the *current* step.

This is the agent equivalent of **CPU branch prediction**. Modern CPUs execute instructions speculatively before knowing which branch a conditional will take. We do the same: execute the most likely next tool call before the model decides. If we're right (~70% hit rate for common patterns), the user sees instant results. If we're wrong, we discard the speculative result silently.

#### How We Build It

1. **Pattern database** — Analyze historical tool-call sequences to build a transition probability matrix:
   ```
   P(google_calendar_create | google_calendar_list) = 0.45
   P(gmail_send | gmail_search) = 0.62
   P(memory_store | memory_search) = 0.38
   ```

2. **Threshold** — Only speculate when P(next_tool | current_tool) > 0.5 (configurable).

3. **Read-only speculation** — Only speculate on **read** operations (list, search, fetch). Never speculatively execute writes (send, create, delete). This eliminates rollback complexity entirely.

4. **Connection warming** — Even when we don't fire the speculative call, we can pre-establish TCP/TLS connections to the likely next API endpoint. This alone saves 100–300ms per call.

5. **Cancel on mismatch** — If the model calls a different tool than predicted, cancel the speculative request (if still in-flight) and proceed normally.

#### Impact

| Scenario | Before | After |
|----------|--------|-------|
| Calendar list → create event | 400ms API latency | 0ms (pre-fetched) |
| Email search → send email | 300ms API latency | 100ms (connection pre-warmed) |
| Memory search → use result | 50ms search latency | 0ms (pre-searched) |
| Mis-prediction | N/A | Speculative result discarded, no cost |

**Expected hit rate: 60–70%** on common workflows (calendar, email, memory patterns).

#### Files Affected
- `src-tauri/src/engine/speculative/` — New module (predictor, cache, executor)
- `src-tauri/src/engine/agent_loop/mod.rs` — Trigger speculation after tool completion
- `src-tauri/src/engine/tools/mod.rs` — Record tool sequences for pattern learning
- `src-tauri/src/engine/sessions/schema.rs` — `tool_sequences` table for pattern storage

## Cross-Cutting: Security Invariants (All Phases)

These security properties apply across every phase. They are not optional enhancements — they are hard requirements that every phase must satisfy before shipping.

### PII and Secret Redaction in Prompts

When agents communicate with each other, call tools, or output execution plans, they process real user data — API keys, OAuth tokens, email addresses, phone numbers, credentials, financial data. None of this should ever leak into model context unnecessarily.

**The Redaction Pipeline:**

Every piece of data that flows into a prompt — tool results, agent-to-agent messages, plan DAG outputs, memory recalls — passes through a redaction layer before reaching the LLM:

1. **Secret zero-out** — Any string matching known secret patterns is replaced with `[REDACTED]` before injection into context:
   - API keys: `sk-`, `GOCSPX-`, `xoxb-`, `ghp_`, `glpat-`, etc. (regex pattern library)
   - OAuth tokens: `ya29.`, `eyJ` (JWT), bearer tokens
   - Private keys: `-----BEGIN`, hex strings >64 chars
   - Environment variables: anything loaded from `.env` that contains `SECRET`, `KEY`, `TOKEN`, `PASSWORD`

2. **PII detection** — Structured PII is identified and masked:
   - Email addresses → `j***@example.com` (first char + domain visible)
   - Phone numbers → `+1 ***-***-1234` (last 4 digits visible)
   - Credit card numbers → `****-****-****-5678`
   - SSN/national IDs → `***-**-6789`
   - IP addresses in private ranges → passed through; public IPs → `***.***.***.***`

3. **Per-tool redaction policy** — Some tools legitimately need PII (e.g., `gmail_send` needs the recipient email). The redaction layer has a tool-specific allowlist:
   ```
   gmail_send:     allow [email_address] in [to, cc, bcc] fields
   google_api:     allow [email_address] in [body] field
   memory_store:   allow [all] — user explicitly asked to remember
   agent_comms:    redact [all secrets], allow [PII with user consent]
   ```

4. **Prompt scrubbing on output** — Before the model's response is displayed to the user or stored in session history, a post-generation scan catches any secrets the model may have echoed back. If the model repeats a token or key (even partially), it's replaced with `[REDACTED]` in the stored/displayed version.

5. **Plan DAG argument sanitisation** — When the model generates an Action DAG (Phase 0), the plan validator checks every argument value against the secret pattern library. If a tool argument contains a raw secret (e.g., the model tried to pass an API key as a function argument instead of using the vault), the plan is rejected with: `"Security: argument 'api_key' in node 'b' contains a secret. Use vault reference instead."`

**What we already have:** The `security.ts` injection scanner (30+ patterns) and `credential_activity_log` table provide the foundation. The redaction pipeline extends these from "detect and warn" to "detect and zero-out before the model ever sees it."

### Encrypted Inter-Agent Communication

When Agent A delegates to Agent B — or when agents exchange messages in a squad, orchestrator, or Tesseract cell — the data in transit and at rest must be encrypted. This becomes especially critical with binary IPC (Phase 3), where human-readable inspection is no longer possible.

**Encryption architecture:**

1. **In-transit encryption (agent-to-agent)** — All inter-agent messages are encrypted with AES-256-GCM before being written to the `agent_messages` table or passed through the binary IPC bus:
   - Encryption key: derived from the app's master key in the OS keychain (same key that protects credentials today)
   - Each message gets a unique 12-byte random nonce
   - The ciphertext + nonce + tag are stored; plaintext never touches SQLite
   - Decryption happens in-memory only, at the moment the receiving agent's context is being built

2. **Ephemeral context keys** — For Plan DAG execution (Phase 0) and Tesseract cells, a per-execution ephemeral key is generated:
   - The plan executor generates a random 256-bit key at plan start
   - All intermediate results (tool outputs, partial state) are encrypted with this key in memory
   - When the plan completes and results are synthesised, the ephemeral key is destroyed
   - This means if the process crashes mid-plan, intermediate results are unrecoverable — by design

3. **Memory encryption** — Engram already encrypts sensitive memory fields with AES-256-GCM. The new phases inherit this:
   - Tool embeddings (Phase 2): not encrypted (tool descriptions are not sensitive — they're the same for all users)
   - Speculative cache (Phase 4): encrypted at rest if it contains user data; speculative results for read-only queries (calendar events, email subjects) are encrypted with the master key before caching

4. **Binary protocol encryption (Phase 3)** — When we move to binary IPC:
   - MessagePack payloads between frontend and engine: encrypted with a per-session key negotiated at app startup (Tauri IPC is already process-local, but defence-in-depth)
   - Protobuf agent-to-agent messages: encrypted at the serialization layer — `encrypt(serialize(message))`, not `serialize(encrypt(message))`. This ensures the binary format itself doesn't leak structure.

5. **Zero plaintext secrets in logs** — The logger (already implemented in `logger.ts` / Rust `tracing`) is extended with a redaction formatter that scrubs secrets from all log output, including debug and trace levels. A secret that appears in a tool result, agent message, or plan argument is never written to disk in plaintext — even in development mode.

**Audit trail:** Every agent-to-agent message exchange is logged in `credential_activity_log` with: timestamp, sender agent ID, receiver agent ID, action type (`delegate`, `broadcast`, `respond`), and whether encryption was applied. The message content itself is NOT logged — only metadata.

## What This Is NOT

To be clear about what we're *not* building:

- **Not "digital telepathy"** — Models don't exchange raw hidden states. No model's internal representation is compatible with another's. This is an unsolved research problem, not an engineering task.

- **Not "latent space transmission"** — The tokenizer/detokenizer is not the bottleneck. It takes microseconds. The transformer forward pass takes seconds. Skipping the tokenizer saves 0.01% of wall-clock time.

- **Not "direct state injection"** — We're not letting AI modify variables without going through our safety stack. The Foreman's API-level monitoring, OAuth scope restrictions, and human-in-the-loop approval all remain in place.

- **Not a new internet protocol** — Gmail, Slack, and GitHub expose REST APIs. We talk to them in REST. Internally, between our own components, we optimize. But we don't pretend the world will rewrite its infrastructure.

What this *is*: removing unnecessary work. Fewer inference calls. Fewer parse failures. Faster internal communication. Smarter tool selection. Each improvement is measurable, reversible, and grounded in proven computer science (DAGs, constrained grammars, vector search, binary serialization, speculative execution).
